# 2. Field Families

| Type | Name      | Description               |
| ---- | --------- | ------------------------- |

## 2.3 Currency pairing convention

DotTalk++ enforces a **paired currency model**:

```text
<AMOUNT_FIELD>   Y
<AMOUNT_FIELD>_CU   C(3)
```

Example:

```text
AMOUNT      Y
AMOUNT_CU   C(3)
```

### Rules

* `_CU` stores a **3-letter currency code**
* automatically normalized to uppercase
* validated independently but logically tied to the `Y` field

### Important constraint

Because DBF field names are limited:

* `_CUR` may truncate â†’ `_CU`
* `_CU` is the canonical suffix

# 4. Field Validation Rules

## 4.1 Date (`D`)

### Accepted forms

* `YYYYMMDD`
* `TODAY`
* `MM/DD/YYYY`

### Canonical storage

```text
YYYYMMDD
```

### Validation rules

* must be 8 digits after normalization
* month/day ranges enforced
* leap year supported

---

## 4.2 Logical (`L`)

### Accepted values

True:

* `T`, `.T.`, `TRUE`, `Y`, `1`

False:

* `F`, `.F.`, `FALSE`, `N`, `0`

### Canonical storage

```text
T or F
```

---

## 4.3 Numeric (`N`)

### Rules

* optional sign
* digits only
* optional single decimal point
* decimals â‰¤ field decimals
* total width â‰¤ field length

### Rejected

```text
abc
123abc
1.2.3
```

---

## 4.4 Float (`F`)

Same structural rules as `N`, but:

* used for floating values
* error message: *invalid float*

### Key rule

No partial parsing allowed:

```text
0.12.5 â†’ reject
```

---

## 4.5 Int32 (`I`)

### Rules

* strict integer only
* no trailing characters
* must fit int32

### Rejected

```text
123abc
1.5
```

---

## 4.6 Double (`B`)

### Rules

* strict floating parse
* must be finite
* no trailing junk

---

## 4.7 Currency (`Y`)

### Rules

* numeric input only
* scaled to 4 decimal places
* stored as fixed-precision value

### Example

```text
88.125 â†’ 88.1250
```

---

## 4.8 Currency Code (`_CU`)

### Rules

* exactly 3 characters
* alphabetic
* auto-uppercase

### Accepted

```text
usd â†’ USD
eur â†’ EUR
```

### Rejected

```text
US
USDX
12$
```

---

## 4.9 Memo (`M`)

### Behavior

* not validated as text
* stored via memo backend
* DBF field stores object ID

### X64 memo detection

* version check
* field width = 8 bytes

---

## 4.10 Character (`C`)

### Current state

* permissive
* length-limited only

### Future direction

* optional validation rules
* pattern/range support

# 5. Command-Surface Behavior

## 5.1 MULTIREP

* first command upgraded
* baseline for validation system

### Behavior

* strict validation
* normalization applied
* no partial coercion

---

## 5.2 REPLACE

* brought to parity with MULTIREP

### Behavior

* same validation rules
* same normalization

---

## 5.3 CALCWRITE

### Special case

CALCWRITE required refactoring because:

* originally used coercive formatting
* evaluated expressions before validation

### Current model

* builds candidate value from expression
* normalizes per field type
* validates strictly
* rejects malformed input

---

### Known subtlety (recent fix area)

Expression parser may partially interpret malformed literals.

Example:

```text
0.12.5 â†’ 0.12 (incorrect)
```

Fix:

* direct literal path for `N/F/D/L`
* bypass partial evaluation
