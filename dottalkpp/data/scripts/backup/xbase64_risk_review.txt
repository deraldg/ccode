1. `0x64` tables can be misrepresented as having fewer records than they actually do, with no way for callers to detect truncation. `[xbase_64.hpp](D:\code\ccode\include\xbase_64.hpp#L105)` clamps `record_count` to `INT32_MAX`, while `[xbase.hpp](D:\code\ccode\include\xbase.hpp#L141)` and `[xbase.hpp](D:\code\ccode\include\xbase.hpp#L143)` expose only 32-bit `recCount()/eof()/bof()` semantics. Any scan, bounds check, or append logic built on `DbArea` can silently behave incorrectly for real large tables.

2. VFP field metadata is parsed and then mostly discarded, which makes the public model internally inconsistent for VFP tables. `[xbase_vfp.hpp](D:\code\ccode\include\xbase_vfp.hpp#L20)` defines `VfpFieldExtras`, but `[xbase_vfp.hpp](D:\code\ccode\include\xbase_vfp.hpp#L156)` only returns it through the loader call; `DbArea` has nowhere to retain it. That means nullable/binary/autoincrement semantics are not part of the persistent table state, even though the loader knows about them.

3. The VFP loader explicitly ignores autoincrement details while advertising support for VFP autoinc variants. `[xbase_vfp.hpp](D:\code\ccode\include\xbase_vfp.hpp#L35)` includes `VfpAutoInc = 0x31`, but `[xbase_vfp.hpp](D:\code\ccode\include\xbase_vfp.hpp#L210)` hard-codes `autoincrement = false`, `next_autoinc = 0`, and `step_autoinc = 0`. If the writer or higher layers assume `0x31` means autoincrement is preserved, they will be wrong.

4. There are multiple “sources of truth” in `DbArea` despite comments claiming a canonical model. `[xbase.hpp](D:\code\ccode\include\xbase.hpp#L115)` says runtime truth is canonical, but the class still keeps legacy aliases `[xbase.hpp](D:\code\ccode\include\xbase.hpp#L241)` and a separate version byte `[xbase.hpp](D:\code\ccode\include\xbase.hpp#L245)` apart from `_hdr.version`. That creates drift risk unless every setter and open/close path updates all mirrored fields consistently.

5. VFP fields are normalized into classic `FieldRec` with lossy conversion. `[xbase_vfp.hpp](D:\code\ccode\include\xbase_vfp.hpp#L197)` zeroes a classic `FieldRec`, truncates names to 10 bytes at `[xbase_vfp.hpp](D:\code\ccode\include\xbase_vfp.hpp#L201)`, and drops displacement/flags/workarea data by setting `field_data_address = 0` at `[xbase_vfp.hpp](D:\code\ccode\include\xbase_vfp.hpp#L205)`. If any downstream logic relies on `_rawFields` for rewrite fidelity or schema round-tripping, VFP tables will lose information.

6. Header interpretation is dialect-specific, but the shared storage type hides that fact. `[xbase.hpp](D:\code\ccode\include\xbase.hpp#L210)` stores all headers in `HeaderRec`, while VFP and `0x64` headers are parsed into different structs and then projected into only a subset of that shape in `[xbase_vfp.hpp](D:\code\ccode\include\xbase_vfp.hpp#L138)` and `[xbase_64.hpp](D:\code\ccode\include\xbase_64.hpp#L89)`. That is workable for read compatibility, but it means `DbArea` cannot faithfully represent the actual on-disk header for those dialects.

7. There is a header-level `clear()` helper with internal linkage that calls `std::system("cls")`, which is unrelated to DBF state and easy to collide conceptually with other `clear` APIs. `[xbase.hpp](D:\code\ccode\include\xbase.hpp#L57)` puts this into an unnamed namespace inside a public header. It won’t violate ODR because of internal linkage, but it is still surprising API pollution in every translation unit that includes the header.

Overall, the design is coherent as a compatibility layer, but not yet as a faithful multi-dialect table model. The main pattern is “parse richer formats, collapse them into the classic runtime shape, and keep just enough extra fields to compile.” That is fine short-term, but it becomes risky once callers start expecting full VFP or true 64-bit behavior.
