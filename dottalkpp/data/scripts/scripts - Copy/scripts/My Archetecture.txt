## What you should do next (safe progression)

Do **not** refactor the command layer yet. First verify the workspace layer behaves correctly at runtime.

### Quick runtime test

Add a temporary test somewhere safe:

```cpp
dottalk::workspace::WorkAreaManager wam;

std::string err;

wam.select(0);
wam.current_slot_ref()->open_table("customers.dbf", &err);

dottalk::workspace::SchemaWorkspace ws;

ws.capture_from_runtime(wam, &err);
ws.save_file("workspace_test.dts", &err);

wam.close_all(&err);

ws.load_file("workspace_test.dts", &err);
ws.apply_to_runtime(wam, &err);
```

This verifies:

* slot capture
* DBF reopen
* cursor restore
* schema serialization pipeline

without touching commands like `USE` or `SELECT`.

## The first command to migrate

Once that test works, move **SCHEMAS SAVE / LOAD** to the new system.

### SCHEMAS SAVE

Instead of manual state capture:

```cpp
SchemaWorkspace ws;
ws.capture_from_runtime(workarea_manager, &err);
ws.save_file(path, &err);
```

### SCHEMAS LOAD

```cpp
SchemaWorkspace ws;

ws.load_file(path, &err);
ws.clear_runtime(workarea_manager, &err);
ws.apply_to_runtime(workarea_manager, &err);
```

That immediately centralizes workspace persistence.

## Architectural note

Seeing your real `DbArea` header confirms something important:

Your engine already had the **correct internal layering**:

```
DbArea
   ├─ record buffer
   ├─ memo handling
   ├─ index manager
   └─ cursor state
```

The workspace layer we just added sits **above that**, which is exactly where session state belongs.

So this integration required **almost no change to the DB engine**, which is a strong sign the architecture is sound.
