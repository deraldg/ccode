Here is the real analysis of that run, ignoring the explicitly marked known canaries.

## Bottom line

This run does **not** meaningfully test the MCC dataset after the memo section.

The dominant failure is simple and severe:

**`SELECT <alias>` is not selecting by table name.**
Every `SELECT STUDENTS`, `SELECT BUILDING`, `SELECT ROOMS`, `SELECT DEPT`, etc. returned:

`Usage: SELECT <0..31 | <name>>`

But the command clearly did **not** accept the name and did **not** switch areas.

So after the memo section, the current area stayed on **`MEMO_X64`**, and nearly every later “table” test was actually being run against the single-record memo table.

That one defect poisons most of the rest of the regression.

## What failed that matters

## 1. `SELECT <name>` is broken in the regression context

This is the single biggest issue.

Examples:

* `SELECT BUILDING` → usage
* `SELECT ROOMS` → usage
* `SELECT DEPT` → usage
* `SELECT MAJORS` → usage
* `SELECT STUD_MAJ` → usage
* `SELECT TERMS` → usage
* `SELECT COURSES` → usage
* `SELECT TEACHERS` → usage
* `SELECT STUDENTS` → usage
* `SELECT ENROLL` → usage
* `SELECT CLASSES` → usage

Since those failed, the current area remained on `MEMO_X64`.

That means all later operations that were supposed to test MCC tables were actually testing:

* fields: `ID`, `TITLE`, `NOTES`
* record count: `1`
* data: `X641`, memo text, etc.

So when you see:

* `SEEK: unknown field: LNAME`
* `REPLACE: field not found`
* `CALCWRITE: unknown field 'GPA'`
* `LOCATE lname = Taylor` → `Not Located.`
* aggregates on GPA failing
* relation setup failing against `MEMO_X64`

those are mostly **downstream consequences**, not independent primary defects.

### Conclusion:

Before trusting this regression, fix or work around `SELECT <name>`.

---

## 2. Public index workflow is only half-integrated

The public workflow appears to attach a CDX container and tag name, but it is not truly operating against the selected table because the area never changed.

Example:

* `SET INDEX TO students.cdx`
* `SET ORDER TO TAG LNAME`

Then `AREA` reported:

* File: `MEMO_X64`
* Index file: `students.cdx (LNAME)`

That is a nonsense pairing: **MEMO_X64 table with STUDENTS index tag**.

So the order state is being set without validating that the index belongs to the current table, or at least it is being reported that way.

That is dangerous.

### This tells you:

The public order/index attachment surface needs a guardrail:

* verify index/container belongs to current table
* or reject mismatched attachment
* or auto-switch appropriately, though rejection is safer

Right now it can create a misleading state.

---

## 3. Public SEEK/FIND/LOCATE integration is not working

Ignoring the poisoned table selection, the run still shows a real integration gap:

* LMDB utility commands can seek real STUDENTS tags
* public `SEEK` cannot use the current order/tag correctly in this scenario

Examples:

* `SEEK WILSON` → `SEEK: unknown field: LNAME`
* `SEEK Davis` → same
* `SEEK Alex` → `unknown field: FNAME`

That means the public `SEEK` path is still too field-centric or is validating against current DBF fields in a way that breaks indexed seek semantics when using tag order.

Since your dev LMDB tools can find records, the missing piece is not raw index storage. It is the **public command glue**.

## 6. `REINDEX` is still using the wrong file identity

This one is real.

When current order was set to `building.cdx` or `students.cdx`, `REINDEX` still tried to find:

`D:\code\ccode\dottalkpp\data\indexes\MEMO_X64.inx`

That is wrong.

This means `REINDEX` is deriving its target from the current table identity rather than the actual attached order/container state, or mixing stale/current area identity with order identity.

So there is a real bug here independent of the SELECT problem.

---

## 7. `RECORDVIEW` vs `DISPLAY` inconsistency for memo

This is real and worth noting.

For the same `MEMO_X64` record:

* `DISPLAY` showed:

  * `NOTES = third x64 memo after clear`
* `RECORDVIEW` showed:

  * `NOTES = 3`

That strongly suggests `RECORDVIEW` is showing the memo link/object-id/raw stored memo reference, while `DISPLAY` resolves memo contents.

That is inconsistent at the user surface.

You need to decide whether `RECORDVIEW` is:

* a raw-storage inspector, or
* a user-facing field-value display

Right now it behaves like the former while being named like the latter.

---

## 8. TUP / TUPLE / SMARTLIST / LIST memo presentation is inconsistent

You can see this in several places:

* `TUP` often showed shortened memo text
* `DISPLAY` showed full memo text
* `RECORDVIEW` showed `3`
* `LIST` / `SL` showed `3`

This is not necessarily wrong internally, but it is inconsistent enough to confuse users.

The likely pattern is:

* tuple/list code uses normalized/display-short/raw-field value
* display command resolves memo content
* recordview shows stored link

That needs a deliberate policy.

## Most important diagnosis

This run was derailed by one architectural problem:

### **Area selection and command targeting are too fragile**

The harness assumed:

`SELECT STUDENTS`

would move the current area to the already-open STUDENTS work area.

Instead, it did nothing useful and left the session on `MEMO_X64`.

From that point on, the regression became largely a test of how commands fail when pointed at the wrong area.

That is still useful, but it is **not** a Beta-0 proof run.

---

## Recommended priority order

1. **Fix `SELECT <name>`** so it reliably selects an already-open area by alias/logical name.
2. Add a visible sanity assertion in the harness after each select, such as:

   * `AREA`
   * or `SHOW TABLE`
     so poisoned runs are caught immediately.
3. Add guardrails so:

   * `SET INDEX TO students.cdx` cannot quietly attach to `MEMO_X64`
   * or at least emits a warning about table/container mismatch.
4. Fix `REINDEX` so it uses actual active order/container state, not stale table-derived `.inx` identity.
5. Only then rerun the public index / seek / relation / buffer gates.
