Unifying Expression Handling in DotTalk++ CLI Commands
Overview

Many DotTalk++ CLI commands currently handle expressions and filtering using legacy methods that are inconsistent with the new AST-based pipeline. Some commands parse simple conditions (e.g. FIELD OP VALUE) and evaluate them with custom logic or predicate helper functions, while newer commands leverage a central AST (Abstract Syntax Tree) expression engine. This has led to duplicated normalization rules and field access logic across the codebase. The goal is to refactor these commands so that all expression parsing and evaluation goes through the unified AST pipeline (with its normalization layer), using RecordView for data access. In cases where complete replacement is not immediately feasible, the legacy parsing (e.g. predicate_chain logic) should be isolated in a compatibility layer. The following sections review each group of command files, identify current expression-handling approaches and inconsistencies, and recommend focused refactoring steps (before/after) to achieve a consistent AST-based implementation.

Commands Using Single-Predicate Evaluation (List, Delete, Recall, etc.)

Current Handling: Commands like LIST, DISPLAY, DELETE, RECALL (see cmd_list.cpp, cmd_delete.cpp, cmd_recall.cpp) use a very simple predicate parsing. They expect conditions in the form FIELD OP VALUE only. The parsing is often done by splitting the input (sometimes via a helper like parse_scan_options) into a field name, an operator, and a value. Evaluation is then done by calling predicates::eval or similar on the current record. This older predicates::eval routine fetches the field’s raw value (as a string) and compares it to the given literal, with internal logic for type normalization (e.g. converting numeric strings to numbers, uppercase string comparison, etc.). For example, eval_cond in predicate.hpp simply parses "FIELD OP VALUE" and delegates to predicates::eval, assuming a single condition. These commands do not support complex expressions with multiple conditions; any attempt to use AND/OR or functions would not be handled by the simple parser. String comparisons are case-insensitive by default in this legacy path (the code uses case-insensitive field lookup and uppercase comparisons). There is also redundant trimming/normalization – e.g. the command parser may trim input, and predicates::eval again normalizes values (removing quotes, spaces, etc.). Field access is inconsistent: they call xbase::DbArea.get() to retrieve a field as a string, then manually trim or convert it depending on type. This means each command re-implements parts of normalization (e.g. trimming spaces for char fields, parsing numbers from numeric fields, interpreting dates) which can lead to subtle differences.

Refactor Plan: Unify these commands to use the central AST pipeline for any filtering or conditional expressions. Instead of parsing into a field/op/value triplet and calling predicates::eval, the command should compile the entire condition string with dottalk::expr::compile_where(). This uses the AST parser to handle complex expressions (multiple conditions, functions, parentheses) uniformly. Next, run the AST through the normalization pass (to ensure types align – e.g. numeric literals and field types are resolved appropriately) and then evaluate it for each record using the AST evaluation interface. In practice, the command loop would do something like:

If a filter expression was provided (e.g. after FOR or implicitly), call compile_where(condStr) to get a compiled expression AST.

For each record, use Expr::eval() (or a typed variant like eval_bool) with a RecordView of the current record. The RecordView (from expr::glue_xbase) will handle retrieving field values in a normalized way. All runtime checks (string comparisons, numeric comparisons) then happen via the AST’s logic.

Before → After Highlights:

Parsing: Replace ad-hoc string splitting with the AST parser (supports full boolean expressions, not just one condition).

Evaluation: Instead of predicates::eval, use program->eval(recordView) where program is the compiled AST. This routes through the unified engine.

Normalization: Eliminate manual trim/case-manipulation in commands – rely on the AST’s normalization layer. For instance, the AST’s NormalizePass and RecordView will ensure a numeric field “1,234.50” is seen as 1234.50, dates “YYYYMMDD” parsed as number, logical “T/F” to 1/0, etc. Case-insensitive string compares should be handled by either uppercasing in the AST or via a unified toggle (see below), rather than each command doing std::toupper on its own.

Field Access: Use dottalk::expr::RecordView (via glue_xbase::make_record_view) for all field reads, instead of calling DbArea.get() and converting. RecordView centralizes access: it provides raw field text for string expressions and numeric conversion for numeric contexts, following DBF normalization rules. This ensures consistency (e.g. right-trimming padded spaces for char fields, interpreting logical fields uniformly) across all commands.

By doing this, commands like LIST/DELETE will be able to support more complex WHERE conditions (multiple AND/OR clauses, functions like UPPER(), etc.) out-of-the-box, and they’ll all behave consistently. The custom helper predicates::eval and related one-off logic can be retired. Case sensitivity can be managed by the AST globally (for example, via a flag or by requiring explicit UPPER()/LOWER() usage), rather than hidden in a global in predx as it is now.

Commands with Custom Expression Parsing (COUNT, FIND, LOCATE)

Current Handling: Some commands implemented their own mini expression parsers or special-case logic:

COUNT (cmd_count.cpp): The COUNT command supports a boolean condition after FOR or WHERE, and even an implicit expression (e.g. COUNT <expr> as shorthand for COUNT FOR <expr>). In the current implementation, COUNT contains a “minimal expression engine” coded directly in its source. The code defines its own Lexer/Parser for conditions supporting basic logical operators AND/OR/NOT and comparison ops (= <> <= >= etc.), with case-insensitive string matching and a few keywords (like DELETED() function). This means COUNT essentially duplicates functionality that the AST should handle – but in a limited way (no parentheses or complex function support, only the specific features coded). This duplication also has its own normalization rules (e.g. always uppercase strings for compare, interpreting empty as false, etc.), which might differ slightly from others.

FIND (cmd_find.cpp): The FIND command doesn’t use a general boolean expression; it implements a specific syntax (FIND <value> IN <field> or <field> = <value>) to search for a substring in a single field. Internally it likely uppercases the search term and field values for case-insensitive matching and scans sequentially. This is an ad hoc parser tied to the command’s purpose.

LOCATE / CONTINUE (cmd_locate.cpp, cmd_continue.cpp): These commands search for the first record meeting a condition. The code uses a utility to extract a FOR <expr> clause or, if FOR is omitted, treats the entire input as an expression (a “triplet shorthand”). It then calls predx::eval_expr to evaluate the condition on each record. The predx::eval_expr function is a separate legacy evaluator that can handle a single condition with simple operators, and it has limited support for functions (it allows UPPER(...) or LOWER(...) on the field side). However, like the others, it’s not a full parser for complex logic. It also relies on global state for case sensitivity. So LOCATE can handle one condition (case-insensitive by default, unless SET CASE was toggled) but would not correctly handle compound expressions with AND/OR (the code would treat them as part of a string literal or fail to parse them).

These custom approaches create redundancy and potential inconsistency. For example, COUNT’s engine might handle NOT or DELETED() differently than how the new AST would, and predx vs. predicate chain might disagree on certain edge cases or require separate maintenance.

Refactor Plan: Each of these should be refactored to use compile_where + AST evaluation, similar to the first group, with any special behaviors incorporated into the AST framework:

COUNT: Instead of using its internal parser, feed the condition string (after FOR/WHERE or the implicit expression) into compile_where(). The AST can naturally handle logical operators (AND/OR/NOT) and comparisons. The few special cases in COUNT can be mapped to AST equivalents: e.g. the DELETED() function can be handled by the RecordView (the RecordView already exposes a pseudo-field “DELETED” returning T/F, or we can introduce a built-in function/node for it). Case-insensitivity is already default in comparisons (unless we add a case-sensitive mode). By using AST eval, COUNT will support parentheses and more complex conditions if needed, beyond its previous “minimal” grammar. The outcome (the count total) remains the same, but the implementation becomes simpler and more robust. We should remove the custom COUNT parser code entirely after verification. All normalization (trimming strings, numeric conversion) will be done by the AST’s passes and RecordView, so the result will be consistent with other commands.

FIND: We can reinterpret FIND’s functionality in terms of the AST. Essentially, FIND value IN field or field = value (with substring semantics) can be seen as a boolean expression field $ "value" (if we treat $ as a “contains” operator, as supported in DotTalk++). The refactor would parse the user’s input to determine the field name and search value (this part is straightforward since the syntax is fixed), then construct or compile an equivalent expression. For example, if the user says FIND "Smith" IN NAME, we compile the expression NAME $ "Smith" using compile_where. Then we simply loop through records and use exprAST->eval(recordView) to test each record. This leverages the AST’s substring operator implementation (and any normalization it does, e.g. it might uppercase both sides if case-insensitive by default). This change would unify FIND’s behavior with general expression handling – e.g. it would automatically respect the global case sensitivity setting (if integrated) or allow more complex find expressions in future. The sequential search logic and output (found/not found) can remain the same, but the evaluation becomes one-liner via AST instead of custom std::string searching. Note: We should ensure the AST’s $ operator is defined to do case-insensitive substring by default (matching current behavior). If not, we might apply an upper() function in the expression or rely on the normalization pass to handle it.

LOCATE/CONTINUE: These should also use compile_where. In the new approach, we can simplify input parsing: if the user provided an explicit FOR <expr>, we already isolate the <expr> string; if they did not, we can take the remainder of the command line as the expression (the “implicit FOR”). In either case, pass that condition string to the parser. The AST will handle any complexity in the expression. During record scanning, use exprAST->eval_bool(recordView) for each record to test the condition. This replaces the call to predx::eval_expr. The benefits are similar: more powerful expression support and a single source of truth for how expressions are evaluated. The predx module (with its own case toggle and limited functions) can be deprecated. One thing to preserve is the behavior of CONTINUE using the last LOCATE’s condition – with AST, we can store the compiled Expr object from LOCATE and reuse it in CONTINUE, rather than re-parsing the text. This would be efficient and clean.

By refactoring these, we eliminate the custom parsing logic in COUNT and FIND and the predx dependency in LOCATE. All these commands will then uniformly respect the same expression syntax and normalization rules. Any necessary unique behavior (like FIND’s default substring search) can be achieved by constructing the appropriate AST (or perhaps providing a convenience in the parser for IN <field> to translate to $ operator).

Data Update Commands (REPLACE, APPEND, etc.)

Current Handling: Commands that write or modify data, such as REPLACE (and the multi-field variant REPLACE... MULTIPLE in cmd_replace.cpp / cmd_replace_multi.cpp), also involve expression logic but in two ways: a conditional filter (e.g. FOR <expr> clause to only replace in certain records) and the new value expressions for fields (the right-hand side of “WITH”).

In the current implementation, the FOR condition in REPLACE likely uses the same simplistic approach as LIST/DELETE – i.e. it may parse a single condition into fld/op/val and call predicates::eval for each record, or ignore complex conditions. (If cmd_replace.cpp doesn’t explicitly handle FOR, it might rely on an existing filter or simply not support complex conditions yet – but this would be a known limitation). There is evidence of duplicated logic; for example, older code might have manually parsed the command to find the “FOR” token and split out the condition string. No unified AST usage is apparent in the current replace code.

For the assignment part (FIELD = <expression> after WITH), it’s unclear if the current code supports anything beyond literal values or simple arithmetic. FoxPro allows expressions on the right-hand side, but DotTalk++ may not fully support that yet in the old code. Possibly, cmd_replace only accepted a literal or one field reference. If any expression was allowed, the implementation might have used the predx::eval_expr or another evaluator per record to calculate the new value. The presence of an EVALUATE or CALC command and an evaluate.cpp suggests that evaluating arbitrary expressions was being developed (perhaps for use in replacement or scanning). There is also field_codecs.cpp which likely handles converting typed values to the DBF field format (e.g. formatting a double to a numeric field string). Currently, data-type normalization for writing might be handled ad-hoc: e.g. if replacing a numeric field, the code might convert a string to double or vice versa and then format to the field width, potentially duplicating logic that exists elsewhere.

Refactor Plan: Unify REPLACE and similar data-modifying commands through the AST for both filtering and value calculation, and use centralized normalization for writing values:

Filter condition: As with previous groups, switch to compile_where for parsing the FOR <expr> clause (if present). This yields an AST (expected type bool). Use Expr::eval_bool(recordView) on each record to decide if the replacement should apply. This replaces any use of predicates::eval or manual condition checks in these commands. It will allow complex conditions in updates (e.g. REPLACE salary WITH 0 **FOR** (salary < 0 OR dept == "HR")). All the type handling (string vs number, etc.) in the condition is handled by the AST layer uniformly.

Value expression: Introduce use of the AST for the right-hand side of assignments. We can extend the expression parser to parse the expression after WITH (this might simply be the same parser, since it can parse any expression, not just booleans). For each target field to replace, compile the provided expression once (outside the record loop). Then for each record, evaluate it (via eval_number, eval_string, etc. depending on field type or expression type). The AST evaluation will produce a result in a high-level type (perhaps a variant or specific type classes). We then normalize that result into the field’s format using a central utility. For numeric fields, for example, if we get a double result from the AST, we should convert it to the proper string (with the field’s width and decimals) using a shared routine (perhaps functions in field_codecs.hpp). For date fields, if the AST returns a date or a number representing YYYYMMDD, format it to "YYYYMMDD". For logical fields, if AST returns a bool or numeric, convert to 'T'/'F'. By routing all these through a single normalization layer, we avoid having each command implement formatting. The value_normalize and xbase_field_getters/putters utilities can likely be extended or reused for this purpose (they currently handle input normalization; similar principles apply in reverse for output).

Use RecordView for context: When evaluating the right-hand side expression for each record, use the same RecordView of the current record so that any field references in the expression (e.g. WITH salary * 1.1 references the current record’s salary) are resolved correctly. The AST already supports field references via RecordView callbacks. (If the expression is just a literal, it will ignore the record data anyway.)

Before → After Example: Suppose the command is REPLACE AGE WITH AGE + 1 FOR GENDER = "F". Currently, this might be handled by retrieving AGE and GENDER via DbArea.get(), converting strings to numbers in C++ manually, checking the condition with custom logic, then writing back. After refactor, it would work as: compile GENDER = "F" to an AST (bool) and AGE + 1 to an AST (numeric expression). For each record: evaluate the condition AST with recordView; if true, evaluate the value AST with recordView; take the numeric result and pass it to a unified writer that updates the AGE field in the DBF (ensuring proper formatting). This would drastically simplify the code in cmd_replace.cpp and guarantee that any expression allowed in the CALC/EVALUATE commands is also allowed in REPLACE. It also means if we implement new functions or operators in the AST, REPLACE automatically gains the ability to use them in the WITH clause.

Append/Import: Similarly, if commands like APPEND FROM or IMPORT allow a WHILE condition or expressions mapping, those should also be converted to use compile_where for conditions and AST for any expressions during import. If currently there is any normalization logic (e.g. reading CSV values and converting to DBF types), that could be aligned to use the same value_normalize routines or field_codecs as well.

In summary, refactoring data update commands to the AST pipeline ensures that filtering and value calculations use the same expression language. It removes ad-hoc conversions; for example, rather than each update command deciding how to treat an empty numeric or how to uppercase a string literal, the normalization layer (which knows that for numeric fields a blank means null, etc.) will handle it consistently.

New AST-Based Commands and Mixed Approaches (SMARTLIST, WHERE, CALC, EVALUATE, BOOLEAN)

Some newer or more advanced commands were introduced specifically to leverage the AST engine. For example, SMARTLIST is described as a LIST-like output but with “SQL-grade filtering”【92†Source】, indicating it supports complex WHERE expressions (and indeed, cmd_smartlist.cpp uses compile_where internally). WHERE is a command to manage the current filter state, which in the implementation uses the new expression pipeline (compiling the filter expression and storing it in a filter registry). CALC (or CALCULATE) and EVALUATE are utility commands to compute expressions on the fly, and BOOLEAN evaluates a logical expression on the current record. These commands partially demonstrate the AST system, but there are still some inconsistencies/legacy usage around them:

SMARTLIST: Currently, SMARTLIST uses the AST for filtering (it calls compile_where(expr) on the provided filter condition and then uses NormalizePass and evaluation). This allows it to accept complex boolean expressions after FOR. However, SMARTLIST might still contain some legacy support or checks. For instance, the code comments refer to “SMARTLIST-style predicate chaining”, and it’s possible the implementation tries to detect simple predicates and evaluate them via a faster path. In any case, since SMARTLIST’s purpose is to showcase the “SQL-grade” capabilities, it should fully commit to the AST. Recommendation: Ensure SMARTLIST exclusively uses compile_where for its filtering. Remove any usage of the old predicate_chain or manual parsing. If there is a debug mode or performance concern that previously led to using predicate chaining (perhaps to quickly handle simple conditions), consider benchmarking the AST; if needed, the predicate chain can be called as an optional fallback (more on that in Legacy Compatibility below). SMARTLIST should also use RecordView for field access when printing results, to be consistent (it likely already uses something similar to format output).

WHERE (Set Filter): The SET FILTER/WHERE commands have been updated to use a centralized filter registry with AST. The filter registry code already calls compile_where and stores the compiled program【70†】. This is a good design: it means the active filter on the workarea is an AST that is applied to each record globally. One thing to check is that older SET FILTER TO might have had a simpler implementation that directly used predicate_chain or predx. If any vestige of that exists, it should be removed. Recommendation: Continue using the AST for filters. To unify, make sure that any command that respects a filter (e.g. LIST by default, if no FOR given, should honor the active filter) also uses the AST representation. Since filter state is stored (possibly as an AST in the registry), ensure commands query the registry’s compiled expression rather than maintaining their own parse. This way, turning a filter on/off is just enabling or skipping an AST check per record. No separate legacy logic needed.

CALC / EVALUATE: The CALC command (and its alias EVALUATE) appears to evaluate an expression for the current record or possibly compute something across records. From the code, cmd_calc.cpp mentions support for tuple results ($N syntax), type checking, and predicate chaining for conditions. This suggests that CALC may allow an optional condition (like CALC <expr> FOR <cond>) or may chain multiple expressions. It currently uses a mix: it attempts try_eval_predicate_chain on the expression (likely to quickly handle simple boolean chains) and if that fails (meaning the expression is more complex or not just a boolean combination), it falls back to using compile_where【89†】. Recommendation: Simplify CALC to use the AST uniformly. If CALC’s purpose is to evaluate an expression (or multiple expressions) on each record, we can break it into two parts: if it has a condition part, use AST for that; for the main expression part, also use AST. The predicate_chain fast-path can be removed or limited to truly legacy fallback. Essentially, we want CALC to behave consistently with how a user would expect an expression to be evaluated in DotTalk++ (which is via the central engine). Any special tuple or multi-result logic can likely be handled by extending the AST (e.g. if $N denotes outputting multiple fields, perhaps the parser can interpret it or we handle it post-evaluation). Removing predicate_chain here will ensure CALC’s behavior (especially in edge cases) matches that of SMARTLIST or other expression uses. All normalization (like ensuring string comparisons in CALC are case-insensitive by default) will then come from the same source.

BOOLEAN: The BOOLEAN command simply evaluates a boolean expression on the current record and prints the result (.T. or .F.). It already uses dottalk::expr::compile_where and then program->eval(rv) on a RecordView【30†】. However, the implementation of RecordView in BOOLEAN was manual: it reads the DBF file directly to get field values and trims them, rather than using the provided glue_xbase::make_record_view. This is an inconsistency in field access – probably done as a quick implementation. Recommendation: Refactor cmd_BOOLEAN.cpp to use make_record_view(area) from glue_xbase instead of custom file I/O. The RecordView from glue will properly handle field lookup (including special fields like DELETED) and numeric conversion per the standard rules. This change would remove a chunk of custom code (the lambdas that open the file and read bytes). If any differences result (for example, the custom code trimmed both leading and trailing spaces for fields, whereas make_record_view might not trim trailing spaces for get_field_str by design), we should standardize on the glue_xbase behavior and adjust via the normalization pass if necessary (i.e. ensure the AST comparator ignores trailing pad spaces for char fields – typically by trimming in normalize_for_compare).

In summary, these AST-centric commands mostly need minor adjustments to fully eliminate ad-hoc code. They should consistently use the common facilities:

Always obtain a RecordView from the DBF via the central glue (rather than manual field reads).

Avoid any direct calls to legacy predicate evaluation. (For instance, if SMARTLIST or CALC currently call try_eval_predicate_chain or predx for any reason, phase that out).

Use eval_bool, eval_number, or eval_string as appropriate for clarity. (The AST Expr class likely has overloads or we can determine type; ensuring we call the correct one will enforce type checking. E.g., eval_bool will throw or error if the expression isn’t boolean, which is useful for commands that specifically need a boolean result.)

Isolating Legacy Predicate Logic (Compatibility Layer)

After refactoring the individual commands, the aim is that all normal expression handling goes through one code path. However, DotTalk++ may want to keep the old predicate logic around temporarily for compatibility or performance. The predicate_chain.* module and the predx namespace are part of this legacy support. Going forward, these should be encapsulated in a thin layer that can be toggled or eventually removed:

Centralize use of predicate_chain: Currently, some code (like CALC) uses try_eval_predicate_chain to detect if an expression is a simple chain of conditions that can be handled without full AST. Instead of each command deciding this, it would be cleaner to have the AST compilation routine itself attempt the legacy parse only if needed. For example, we could modify compile_where to internally call try_eval_predicate_chain on the input; if it recognizes and evaluates a simple expression successfully, we could return a trivial AST node (or a special value) representing that boolean result. However, this is likely not necessary once AST is robust. A better approach is to provide a separate function (in a compatibility util) that commands can call if they explicitly want to use legacy evaluation for some reason. For instance, a legacy_eval_condition(area, exprStr, bool &result) that wraps try_eval_predicate_chain could be used in debugging or to compare outcomes. But ideally, production code paths shouldn’t need it.

Predicates and Predx as fallback: The old predicates::eval (single condition evaluator) and predx::eval_expr could be kept in a “legacy compatibility” module to support any external scripts or edge cases expecting the old behavior. After refactoring, no internal command should call these directly. If absolutely needed (for example, if SET FILTER wants to detect a very simple filter to optimize something), it could call them internally but that should be avoided unless a measurable performance issue arises. Instead, rely on the AST (which can be optimized with AST passes and cached bytecode if needed in the future).

Testing and parity: During the transition, it might be useful to run tests where the output of the new AST-based evaluation is compared to the old predicate_chain on a variety of expressions to ensure parity (especially around string comparison nuances, empty values, etc.). The legacy layer can assist in this testing. But structurally, isolate all legacy predicate code in one place – e.g. a single source file or namespace (perhaps expr_compat), with clearly marked functions. This makes it easier to remove later. All commands should then call the new pipeline; none should directly reference predicate_chain or predx. If a bug or unimplemented feature in the AST engine forces a fallback, funnel that through a single function that logs the case (so you know what needs improvement in AST).

By implementing these refactors, the DotTalk++ CLI will have a unified expression handling mechanism: every command that filters or evaluates expressions will use the same parser, same AST, and the same normalization rules. This unification will eliminate redundant code (making maintenance easier) and ensure consistency from the user’s perspective. A condition used in one command will behave the same in another. The legacy code will be confined to a compatibility layer that can be turned off once confidence in the AST pipeline is high. Overall, these surgical refactors (replacing predicate-chain and ad-hoc paths with compile_where + NormalizePass + AST eval, and standardizing on RecordView for field data) will greatly streamline the CLI implementation while matching or expanding the capabilities of the original commands.
