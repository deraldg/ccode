## Problems / regressions revealed

### A) Date arithmetic is only partially implemented (or typed incorrectly)

You have **two conflicting behaviors**:

1. Field date arithmetic works:

* `CALC DOB + 1` ‚Üí `19990103` ‚úÖ
* `DATEADD(DOB, 30)` ‚Üí `19990201` ‚úÖ
* `DATEDIFF(DATE(), DOB)` ‚Üí `9883` ‚úÖ

2. But system date arithmetic does *not*:

* `CALC DATE() + 2` ‚Üí `20260123` ‚ùå (should be `20260125`)
* `CALC DATE() - 2` ‚Üí `20260123` ‚ùå (should be `20260121`)
* `CALC DATE() + 30` ‚Üí `20260123` ‚ùå (should be `20260222`)
* `CALC DATE() - 30` ‚Üí `20260123` ‚ùå (should be `20251224`)
* `CALC DATE() - DOB` ‚Üí `20260123` ‚ùå (should be a day count, like DATEDIFF)

**Interpretation:** `DATE()` is being treated as a ‚Äúspecial‚Äù expression node that returns a date-typed value, but the `+/-` operator path is likely:

* refusing to do date+int when the LHS is a function call result (DATE()) but allowing it when LHS is a field (DOB), **or**
* converting the `DATE()` result to a ‚Äústring-ish / non-date scalar‚Äù too early so the operator returns the left operand unchanged, **or**
* falling through to a ‚Äúunsupported op ‚Üí return LHS‚Äù behavior rather than raising an error.

The fact that `DATEADD(DOB, 30)` works suggests your date math helpers are good; what‚Äôs missing is **operator dispatch** consistency for:

* `date + integer`
* `date - integer`
* `date - date` (should be integer days, matching your DATEDIFF)

### B) String builtins are broken in CALC/TUPLE context

You flagged it yourself in the script, and the run confirms it:

* `CALC TUPLE UPPER(LNAME)` outputs blank ‚ùå
* `CALC TUPLE LOWER(LNAME)` blank ‚ùå
* `CALC TUPLE PROPER(LNAME)` blank ‚ùå
* `CALC TUPLE LEFT(LNAME, 3)` returns `.F.` ‚ùå (wrong type entirely)
* `CALC TUPLE RIGHT(FNAME, 2)` returns `.F.` ‚ùå
* `CALC TUPLE ALLTRIM(EMAIL)` blank ‚ùå

**Interpretation:** the ‚ÄúTUPLE expression mode‚Äù inside CALC is not receiving/returning string values correctly. The `.F.` results strongly suggest the evaluator is:

* treating function calls as predicate/boolean nodes in that context, or
* failing argument evaluation (undefined/empty) and defaulting to false, or
* returning a variant type that prints as blank unless it is numeric/boolean.

Given that `TUP` prints string fields fine, and boolean expressions work, the failure is likely localized to the **CALC expression engine‚Äôs string-function registration or value printing path** in ‚ÄúTUPLE ‚Ä¶‚Äù mode.
