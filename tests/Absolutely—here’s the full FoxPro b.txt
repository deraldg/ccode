FoxPro Search Commands: FIND, SEEK, and LOCATE  
A codex-grade breakdown for legacy rigor and teachable overlays

Overview Table:  
FIND – Requires index, searches indexed field only, does not return Boolean, fast performance, fails silently  
SEEK – Requires index, searches indexed field only, returns Boolean (.T./.F.), fast performance, preferred for logic  
LOCATE – No index required, searches any field or condition, returns Boolean (.T./.F.), slower performance, most flexible

FIND:  
- Requires an active index (SET ORDER TO)  
- Searches only indexed fields  
- Syntax: FIND <exp>  
- Behavior: Moves record pointer to match if found, moves to EOF if not found  
- Use Case: Fast lookup when index is guaranteed and value exists  
- Warning: Fails silently if no index is active or field isn’t indexed

SEEK:  
- Requires an active index  
- Searches only indexed fields  
- Syntax: SEEK <exp>  
- Behavior: Returns .T. if found, .F. if not; moves record pointer to match if found  
- Use Case: Ideal for conditional logic and error trapping  
- Example:  
  IF SEEK("Smith")  
     ? "Found!"  
  ELSE  
     ? "Not found."  
  ENDIF  
- Note: More robust than FIND—preferred in codex-grade logic

LOCATE:  
- Does NOT require an index  
- Searches any field, indexed or not  
- Syntax: LOCATE FOR <condition>  
- Behavior: Scans from top of table (or filtered scope); moves record pointer to first match; returns .T. if found, .F. if not  
- Use Case: Flexible search across any field or condition  
- Example:  
  LOCATE FOR LASTNAME = "Smith" AND AGE > 40  
  IF FOUND()  
     ? "Match found"  
  ENDIF  
- Note: Slower on large tables—use with caution in performance-critical paths

Codex Notes:  
- SEEK is your go-to for indexed lookups with Boolean feedback  
- LOCATE is your fallback for raw scans or complex conditions  
- FIND is legacy-fast but brittle—use only when index integrity is guaranteed
